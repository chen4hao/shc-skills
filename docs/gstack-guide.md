## 一、框架簡介

**gstack** 是由 Y Combinator 現任 CEO Garry Tan 開發的開源 AI CLI Skills 集合，將 Claude Code 轉變為一個完整的虛擬工程團隊。包含 **28 個專業化 skills**，涵蓋產品策略、設計、開發、QA、安全、部署、監控到反思的完整軟體開發生命週期。

### 核心哲學：Boil the Lake

> 在 AI 協助下，完整實施的邊際成本接近零，因此應始終選擇**完整方案**而非快速解決方案。

### 驚人產出數據

Garry Tan 在使用 gstack 的 60 天內（同時擔任 YC CEO），達成：
- **600,000+ 行生產程式碼**（其中 35% 為測試）
- **每日 10,000-20,000 行程式碼**

## 三、安裝方式比較：Plugin vs Skill

gstack **目前沒有官方 Plugin 市集版本**，僅提供 Skill（本地安裝）方式。

| 比較項目 | Plugin（外掛市集） | Skill（本地安裝） |
|---------|-------------------|------------------|
| **是否可用** | ❌ 無官方 Plugin | ✅ 開源 MIT 授權 |
| **安裝指令** | N/A | `git clone ... ~/.claude/skills/gstack && ./setup` |
| **更新方式** | N/A | `/gstack-upgrade` 或手動 `git pull` |
| **自訂彈性** | N/A | 極高，可直接修改任何 SKILL.md |
| **平台支援** | N/A | Claude Code、Codex CLI、Cursor、GitHub Copilot |
| **MCP 支援** | N/A | 不依賴 MCP，使用原生 slash commands |
| **離線使用** | N/A | 本地檔案，完全離線可用 |

### 與 Superpowers 的安裝方式比較

| 比較項目 | gstack | Superpowers |
|---------|--------|-------------|
| **Plugin 版** | ❌ 無 | ✅ `claude plugin install superpowers` |
| **Skill 版** | ✅ git clone + setup | ✅ `npx skills add obra/superpowers -g --all` |
| **安裝複雜度** | 中（需 clone + 執行 setup 腳本） | 低（一行 npx 或 plugin install） |
| **自訂彈性** | 極高 | 高 |
| **Skills 數量** | 28 個 | 14 個 |
| **覆蓋範圍** | 全生命週期（含產品策略、部署監控） | 開發流程核心（設計到合併） |
| **特色定位** | 產品導向（CEO 視角）| 工程紀律導向（TDD 視角）|

### 建議

- **重視產品品質與完整生命週期** → gstack（涵蓋策略、設計、部署、監控）
- **重視工程紀律與 TDD** → Superpowers（強制測試優先、系統性除錯）
- **兩者互補** → 可同時安裝，gstack 管產品流程，Superpowers 管開發紀律

## 四、最推薦 Skills 速查表

以下精選 **10 個最實用的 skills**，依開發流程順序排列：

### 1. /office-hours（YC Office Hours）

| 欄位 | 內容 |
|------|------|
| **名稱** | office-hours |
| **最佳用途** | 在動手前用 YC 式提問法釐清需求本質 |
| **流程** | 六個強制問題（實際需求？現狀？迫切細節？最窄楔形？觀察？未來契合？）→ 兩種模式選擇 → 產出設計文件 |
| **實用場景** | 新專案啟動、功能需求模糊、想避免做錯方向 |
| **範例** | 「我想做一個通知系統」→ agent 問：你要解決什麼痛點？現有方案缺什麼？最小可行版本是什麼？→ 產出聚焦的設計文件 |

### 2. /autoplan（自動審查管線）

| 欄位 | 內容 |
|------|------|
| **名稱** | autoplan |
| **最佳用途** | 一個指令完成 CEO + 設計 + 工程三重審查 |
| **流程** | 自動執行 `/plan-ceo-review` → `/plan-design-review` → `/plan-eng-review` → 僅在品味決策時暫停徵詢 |
| **實用場景** | 想要完整審查但不想逐一執行三個指令 |
| **範例** | `/autoplan` → 自動分析產品策略（是否值得做？）→ 設計評分（UI 夠好嗎？）→ 工程審查（架構穩嗎？）→ 僅提出 2-3 個需要人類判斷的問題 |

### 3. /plan-design-review（設計審查）

| 欄位 | 內容 |
|------|------|
| **名稱** | plan-design-review |
| **最佳用途** | 用 0-10 評分量化設計品質，並說明如何達到 10 分 |
| **流程** | 每個設計維度評分 → 說明 10 分標準 → AI Slop 檢測 → 互動式修改計畫直到達標 |
| **實用場景** | UI/UX 品質把關、避免 AI 生成的「千篇一律」設計 |
| **範例** | 「審查我的登入頁面設計」→ 清晰度 7/10（缺少錯誤狀態）、一致性 5/10（字體混用）→ 具體建議如何提升到 9+ |

### 4. /investigate（系統性除錯）

| 欄位 | 內容 |
|------|------|
| **名稱** | investigate |
| **最佳用途** | 遇到 bug 時強制根因分析，避免盲目修補 |
| **流程** | 鐵律：先調查再修復 → 追蹤數據流 → 測試假設 → 失敗 3 次自動停止重新思考 |
| **實用場景** | 生產 bug、測試失敗、非預期行為 |
| **範例** | 「付款流程偶爾失敗」→ agent 追蹤完整請求路徑 → 發現 race condition 在併發場景下丟失 session → 根因修復而非加 retry |

### 5. /review（程式碼審查）

| 欄位 | 內容 |
|------|------|
| **名稱** | review |
| **最佳用途** | 找出通過 CI 但會在生產爆炸的隱藏問題 |
| **流程** | 掃描變更 → 找隱藏 bug → 自動修復顯而易見問題 → 標記完整性差距 → 報告 |
| **實用場景** | PR 合併前、重大功能完成後、部署前最後檢查 |
| **範例** | `/review` → 發現：N+1 查詢在小數據集 CI 通過但生產 10 萬筆會超時 → 建議加 eager loading |

### 6. /qa（自動化 QA）

| 欄位 | 內容 |
|------|------|
| **名稱** | qa |
| **最佳用途** | 自動測試應用、找 bug、修復並生成回歸測試 |
| **流程** | 測試應用 → 發現 bug → 原子提交修復 → 自動產生回歸測試 → 重新驗證 |
| **實用場景** | 功能開發完成後、發佈前品質檢查 |
| **範例** | `/qa` → 發現表單驗證漏洞（空白 email 可提交）→ 修復 → 產生 `test_empty_email_rejected()` 回歸測試 |

### 7. /cso（安全審計）

| 欄位 | 內容 |
|------|------|
| **名稱** | cso |
| **最佳用途** | CISO 等級的全面安全掃描 |
| **流程** | 秘密考古學 → 依賴供應鏈 → CI/CD 管道 → LLM/AI 安全 → OWASP Top 10 → STRIDE 威脅建模 → 主動驗證 |
| **實用場景** | 發佈前安全檢查、月度安全掃描、合規要求 |
| **範例** | `/cso` → 發現：.env.example 含真實 API key 殘留、依賴有已知 CVE、缺少 CSRF token → 逐一修復建議 |

### 8. /ship（發佈）

| 欄位 | 內容 |
|------|------|
| **名稱** | ship |
| **最佳用途** | 一鍵完成從同步到開 PR 的完整發佈流程 |
| **流程** | 同步主分支 → 跑測試 → 審計測試覆蓋 → 推送 → 開 PR |
| **實用場景** | 功能完成後快速發佈、避免忘記跑測試就推代碼 |
| **範例** | `/ship` → 自動 rebase main → 跑全部測試（通過）→ 覆蓋率 87% → 推送 → 建立 PR 附帶變更摘要 |

### 9. /browse（瀏覽器測試）

| 欄位 | 內容 |
|------|------|
| **名稱** | browse |
| **最佳用途** | 用真實瀏覽器自動化測試 UI 行為 |
| **流程** | 啟動 Chromium → 導航頁面 → 點擊互動 → 截圖 → 比較前後差異 → 測試響應式佈局 |
| **實用場景** | 前端 UI 驗證、表單測試、跨裝置測試、視覺回歸 |
| **範例** | `/browse` → 打開登入頁 → 填寫表單 → 提交 → 截圖確認跳轉成功 → 測試手機版面是否正常 |

### 10. /guard（完全安全模式）

| 欄位 | 內容 |
|------|------|
| **名稱** | guard |
| **最佳用途** | 生產環境操作的最大安全防護 |
| **流程** | 啟用 `/careful`（破壞性命令警告）+ `/freeze`（限制編輯範圍）→ 每個危險操作需確認 |
| **實用場景** | 操作生產環境、修改關鍵配置、不熟悉的 codebase |
| **範例** | `/guard src/` → 之後所有編輯限制在 `src/` 內 → 若嘗試 `rm -rf` 會先警告 → 防止意外破壞 |
